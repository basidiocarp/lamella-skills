### Team Pre-Mortem Facilitation Guide

```
## Pre-Mortem Meeting Agenda

**Duration:** 45-60 minutes
**Attendees:** Full project team
**Materials:** Sticky notes, timer, whiteboard

1. Restate the project goal and success criteria
2. Ask everyone to imagine the effort failed badly
3. Capture failure reasons silently for 5-10 minutes
4. Group similar risks into themes
5. Rank the highest-impact failure modes
6. Turn top risks into owners, guardrails, and follow-ups

Facilitator prompts:
- "What are we assuming will go right?"
- "Where are we most dependent on one person or team?"
- "What would make this slip quietly until too late?"

- Capture in doc
- Schedule follow-up
- Thank team for honesty
```

### Pre-Mortem Question Bank

```
## Pre-Mortem Prompts

Use these to stimulate thinking about different failure modes:

### EXECUTION
- What dependency is most likely to be late?
- What step is still too manual?

### DECISION QUALITY
- What assumption has the weakest evidence?
- Where could we optimize for the wrong metric?

### ORGANIZATIONAL RISK
- Which approval path could stall the work?
- What cross-team handoff is least reliable?

### CUSTOMER IMPACT
- What would make the shipped result miss the real user problem?
- Where could support or onboarding fail after launch?

- What should we have done sooner?
- What external event affected us?
- What deadline was unrealistic?
```

### Pre-Mortem vs Post-Mortem

```
## When to Use Each

### PRE-MORTEM (Before)
✓ Project hasn't started yet
✓ Major decision pending
✓ Team can still change approach cheaply

### POST-MORTEM (After)
✓ Incident or failed initiative already happened
✓ Need root cause, learning, and follow-up actions
✓ Timeline evidence is available

Rule of thumb:
- Use a pre-mortem to prevent avoidable failure.
- Use a post-mortem to learn from real failure.

Research shows Pre-Mortem generates 30% more failure reasons.
Why? "Prospective hindsight" makes imagined failure feel real.
```
