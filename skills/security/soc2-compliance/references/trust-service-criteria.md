# Trust Service Criteria

Security is mandatory. The other categories are optional and should match real customer promises.

## Category Guide

| Category | Include It When |
|----------|-----------------|
| Security | Always; it is required for every SOC 2 report |
| Availability | Customers depend on uptime, recovery, or SLA commitments |
| Confidentiality | The service handles customer confidential information beyond normal application data |
| Processing Integrity | The product performs business-critical calculations, transformations, or workflow enforcement |
| Privacy | The service processes regulated or customer-sensitive personal information |

## Security Common Criteria Areas

Use these as the core control families when building the matrix:

- Control environment
- Communication and information
- Risk assessment
- Monitoring
- Control activities
- Logical and physical access
- System operations
- Change management
- Vendor and disruption risk

## Control Mapping Pattern

For each criterion, capture:

- control owner
- control frequency
- system or process used
- evidence expected
- current status

## Evidence-Oriented View

When a criterion is in scope, ask:

1. What control demonstrates this requirement?
2. Who performs or reviews it?
3. What evidence proves it happened?
4. How often should the evidence appear?

## Cross-Framework Use

If the company already uses ISO 27001, internal control matrices, or customer security questionnaires, map those artifacts instead of starting from a blank sheet. The goal is a usable SOC 2 control matrix, not duplicate documentation.
