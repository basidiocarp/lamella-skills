# Company Operating Rhythm

Use this skill to define how a company runs week to week and quarter to quarter. It is useful when ownership is fuzzy, meetings drift, priorities multiply, or strategy fails to translate into execution.

## When to Use

- Selecting a management rhythm for a growing team
- Clarifying accountability across functions
- Designing or repairing weekly and quarterly planning cadence
- Comparing EOS, Scaling Up, OKRs, or a custom hybrid

## Workflow

1. **Start with the operating problem**
   Identify whether the main issue is ownership confusion, meeting sprawl, weak scorecards, poor issue resolution, or slipping quarterly priorities.

2. **Pick the minimum viable operating model**
   Choose the lightest system that solves the current problem. Use `references/os-comparison.md` when framework choice is the main question.

3. **Define the backbone**
   Build:
   - an accountability chart
   - a weekly scorecard
   - a leadership meeting cadence
   - an issue resolution loop
   - a quarterly priority format

4. **Design the rollout**
   Introduce the rhythm in phases rather than all at once. Use `references/implementation-guide.md` to stage the first 90 days.

5. **Document the rules clearly**
   Specify who owns each function, what gets reviewed weekly, how issues are solved, and how quarterly priorities are declared done.

Use `references/all-hands-template.md` when the user needs a recurring internal update structure that matches the operating rhythm.

## Output Expectations

Produce:
- a recommended operating model
- the core meeting rhythm
- the weekly scorecard structure
- the ownership model
- a 30-60-90 day rollout plan

## Quality Checklist

- [ ] Each key function has one clear owner
- [ ] The scorecard uses a small set of weekly metrics
- [ ] The meeting cadence replaces low-value meetings instead of adding more
- [ ] Quarterly priorities are specific enough to be done or not done
- [ ] The rollout is staged and has an explicit owner
