# Pricing Models

Use this reference when the hardest part is choosing how pricing should scale.

## Model Selection

| Model | Best fit | Strength | Failure mode |
|-------|----------|----------|--------------|
| Per seat | Collaboration tools and team software | Revenue grows with headcount | Weak fit when one power user drives most value |
| Usage-based | APIs, infrastructure, AI, communications | Revenue grows with actual consumption | Unpredictable bills create friction and self-capping behavior |
| Feature-tiered | SMB-to-enterprise software with clear plan boundaries | Easy to explain and budget | Weak tiers cluster users into one plan |
| Flat fee | Simple products with stable service cost | Low-friction buying motion | Heavy users are subsidized and expansion stalls |
| Hybrid | Products with both platform value and variable usage | Creates a revenue floor plus upside | Too many moving parts can confuse buyers |

## Selection Questions

1. Does customer value scale with users, usage, features, or outcomes?
2. Which metric is easiest for a buyer to understand and forecast?
3. Which metric is hardest to game or negotiate around?
4. Does the product need an enterprise boundary for compliance, support, or contracts?
5. Is simplicity more important than precise value capture for this market?

## Failure Modes To Check

- Seat pricing for products where one operator does the work and everyone else only looks
- Flat pricing where some customers create 10x the service cost of others
- Feature tiers that separate arbitrary features instead of segment-relevant needs
- Usage pricing with bills that spike unpredictably
- Enterprise tiers that add vague prestige instead of real governance or control

## Tier Heuristics

- Entry tier: enough value to convert the smallest viable segment without losing money
- Middle tier: default option for the intended growth segment
- Top tier: for control, scale, compliance, and support requirements, not cosmetic extras

## Price Increase Questions

- Is the increase tied to more value, a new position in the market, or a cost reset?
- Are existing customers grandfathered, transitioned later, or moved now?
- What should change for new customers first before broader rollout?
- Which accounts need a direct explanation instead of a generic announcement?
