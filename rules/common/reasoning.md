# Reasoning

Trace problems at the right level before jumping to code.

## Levels

- `why`: requirements, domain rules, constraints
- `what`: design, abstractions, tradeoffs
- `how`: implementation details and APIs

## Rules

- start at the level the problem presents itself, then trace up or down as needed
- do not answer a design or requirements problem with a code-only fix
- externalize reasoning for complex multi-step work if the decision path matters more than the final code
