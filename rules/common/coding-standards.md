# Coding Standards

Follow the active project's real standards before applying generic preferences.

## Rules

- prefer the repo's formatter, linter, and existing conventions over invented style
- match the local file's naming, error-handling, and test patterns unless you have a concrete reason to change them
- keep changes aligned with the project's architecture and dependency boundaries
- add comments only where they clarify intent, invariants, or non-obvious tradeoffs
